## Architecture Notes

- Current Spring implementation is Spring MVC + service-layer, not a one-to-one MediatR port.
- The original ASP.NET Core backend used MediatR `IRequest<T>` / `IRequestHandler<T>` extensively.
- Java now uses a direct-handler CQRS style: controllers inject handlers directly, handlers call store/service interfaces, and no mediator/dispatcher library is used.
- `ContentService` still acts as the JDBC-backed adapter for content store interfaces; future refactors should split it into feature-specific store implementations.
- PostgreSQL `timestamptz` values should be mapped through `JdbcData.nullableInstant(...)` instead of `ResultSet#getObject(column, Instant.class)`; runtime smoke caught and fixed the remaining identity/AI paths.

## Remaining Parity Caveats

- Google OAuth2 callback support is implemented through Spring Security OAuth2 Login; production still needs real Google client ID/secret and redirect URI configuration.
- AI/Codex and HLS processing endpoints are deterministic compatibility implementations, not full long-running external provider pipelines yet.
- Full Playwright browser parity now passes against the Docker dev stack in this turn.
- Historical docs under `docs/ralph/**` still describe the previous ASP.NET Core migration history and are intentionally left as historical artifacts.
