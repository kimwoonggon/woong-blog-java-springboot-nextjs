## Backend Refactor Interpretation

The table above should be used as a flow map, not only as a frontend inventory. The `Handler` and `Store/service` columns show which backend areas already follow the Content CQRS split and which areas still route through broad services or direct endpoint logic.

Use this rule while reading rows:

- Rows with `*CommandHandler`/`*QueryHandler` plus `*CommandStore`/`*QueryStore` are already close to the target backend shape.
- Rows with `Direct endpoint-to-*Service calls`, `Mostly direct endpoint logic`, or `Owning mutation handler where persisted content changes; upload endpoint calls service directly` are migration candidates.
- Rows where one behavior spans multiple backend modules should drive integration tests first, then unit tests around extracted handler/policy logic.
- Rows with stable JSON DTOs and frontend API clients are candidates for Pact contract tests.

## Pact Contract Scope

Pact should start with read-heavy, stable JSON contracts:

- `GET /api/public/home`
- `GET /api/public/site-settings`
- `GET /api/public/pages/{slug}`
- `GET /api/public/blogs`
- `GET /api/public/blogs/{slug}`
- `GET /api/public/works`
- `GET /api/public/works/{slug}`
- `GET /api/auth/session`

Provider verification should run against the ASP.NET app on a real local TCP socket with deterministic provider states. Do not use an in-memory `WebApplicationFactory`/`TestServer` host for Pact.NET provider verification, because the verifier must be able to call the provider over HTTP from outside the ASP.NET in-memory server. Dynamic values such as ids, dates, URLs, pagination counts, and optional media fields should use Pact matchers instead of brittle exact values.

Do not start Pact with multipart uploads, HLS/video processing, or AI batch background flows. Those are better covered first with integration and Playwright tests, then narrowed into contracts only after the API shape stabilizes.

## SonarAnalyzer Placement

Do not add SonarQube server/scanner CI for this repository now. Add `SonarAnalyzer.CSharp` as a compile-time analyzer package when backend tests are physically split into unit, integration, architecture, and contract projects.

Recommended minimum rule:

- Run analyzers as part of normal `dotnet test` project builds.
- Keep `SonarAnalyzer.CSharp` package versions aligned across backend test projects.
- Do not gate browser or compose checks on a SonarQube server result.
- Keep generated folders and test artifacts out of source-based architecture scans where applicable.
